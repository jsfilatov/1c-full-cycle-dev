## Принципы работы

- **Адаптивность**: количество агентов и глубина анализа зависят от сложности задачи
- **Ранняя валидация**: ревью плана до реализации, а не после
- **Уточнение требований**: выявление всех неоднозначностей до проектирования через вопросы пользователю
- **Атомарные шаги**: этапы реализации с критериями приемки и проверками
- **Отслеживание прогресса**: после завершения каждой фазы отмечай её как завершённую в списке задач
- **Автоматическая реализация**: выполнение задач из таск-листа без постоянного подтверждения пользователя
- **Контекстное ревью**: проверка кода с учётом PRD, тасклиста и архитектурного плана

## Phase 0: Инициализация и оценка сложности

**Цель**: понять масштаб задачи и создать структуру для работы

Начальный запрос: $ARGUMENTS

**Действия**:
1. Создай список задач со всеми фазами
2. Создай директорию `.tasks/task-[feature-name]/` для хранения артефактов
3. **Оцени сложность задачи** (простая/средняя/сложная/критичная):
   - **Простая**: небольшое изменение, очевидная реализация
   - **Средняя**: затрагивает несколько модулей, требует понимания архитектуры
   - **Сложная**: большая доработка, несколько подсистем, неочевидные решения
   - **Критичная**: архитектурные изменения, влияние на всю систему, высокие риски
4. Запиши оценку сложности в файл `.tasks/task-[feature-name]/phase0-complexity.md`

## Phase 1: Discovery

**Цель**: понять, что нужно построить

**Действия**:
1. Если доработка неясна, спроси пользователя:
   - Какую проблему они решают?
   - Что должна делать доработка?
   - Есть ли ограничения или требования?
2. Резюмируй понимание и получи подтверждение от пользователя
3. Сохрани подтвержденное понимание в файл `.tasks/task-[feature-name]/phase1-requirements.md`:
   - Исходный запрос из $ARGUMENTS
   - Уточняющие вопросы и ответы (если были)
   - Резюме понимания задачи
   - Ключевые требования и ограничения
   - Подтверждение пользователя

## Phase 2: Исследование кодовой базы 1C

**Цель**: понять существующий код и паттерны

**ПРИНЦИПЫ АДАПТИВНОГО ИССЛЕДОВАНИЯ**:

ТЫ ПРИНИМАЕШЬ РЕШЕНИЕ о стратегии исследования на основе:
- Оценки сложности из Phase 0
- Характера задачи (новая доработка vs расширение существующей)
- Того, насколько понятна область кодовой базы

**Ключевой принцип**: глубина важнее ширины - лучше запустить агентов последовательно для углубления, чем параллельно на одно и то же.

**Твои решения**:
1. **Сколько агентов `1c-code-explorer` запустить?** (1-4+)
   - Для простых задач может хватить одного
   - Для сложных может понадобиться несколько с углублением

2. **Параллельно или последовательно?**
   - Параллельно - если нужно изучить разные независимые области
   - Последовательно - если нужно углубление: первый агент находит, второй копает глубже

3. **На какие аспекты фокусироваться?**
   - Похожие доработки и паттерны
   - Архитектура и слои абстракции
   - Интеграции и зависимости
   - Критичные компоненты, найденные предыдущими агентами

4. **Итеративное углубление**
   - Можешь запустить агента, посмотреть результаты, решить нужно ли ещё углубление
   - Каждый следующий агент может фокусироваться на находках предыдущих

**Действия**:
1. Прими решение о стратегии исследования (обоснуй коротко)
2. Запусти агентов `1c-code-explorer` согласно своей стратегии:
   - Передай путь к требованиям: `.tasks/task-[feature-name]/phase1-requirements.md`
   - Укажи фокус исследования для каждого агента (паттерны, архитектура, похожие доработки и т.д.)
3. Каждый агент должен вернуть список нескольких ключевых файлов для чтения
4. Прочитай все указанные файлы для глубокого понимания
5. Оцени: достаточно ли понимания? Нужно ли ещё углубление? Если да - запусти ещё агентов
6. Представь полную сводку найденных паттернов
7. Сохрани сводку в файл `.tasks/task-[feature-name]/phase2-exploration.md`:
   - Найденные паттерны и соглашения
   - Похожие доработки с ссылками на файлы
   - Ключевые компоненты и их назначение
   - Архитектурные инсайты
   - Список всех прочитанных ключевых файлов

## Phase 3: Уточняющие вопросы

**Цель**: заполнить пробелы и разрешить все неоднозначности ДО проектирования

**Действия**:
1. Просмотри находки по кодовой базе и исходный запрос на доработку
2. Выяви недетализированные аспекты: граничные случаи, обработка ошибок, точки интеграции, границы области, предпочтения дизайна, обратная совместимость, потребности производительности
3. **Представь все вопросы пользователю в ясном, организованном списке**
4. **Жди ответов перед проектированием архитектуры**
5. Сохрани вопросы и ответы в файл `.tasks/task-[feature-name]/phase3-clarifications.md`:
   - Список вопросов с контекстом
   - Ответы пользователя на каждый вопрос
   - Принятые решения и рекомендации

Если пользователь говорит "как считаешь нужным", предоставь свою рекомендацию и получи явное подтверждение.

## Phase 3.5: Создание PRD

**Цель**: создать детальный Product Requirements Document (PRD) на основе собранной информации

**Действия**:
1. На основе артефактов Phase 1-3 создай PRD по следующей структуре:
   - **Введение/Обзор**: описание задачи и проблемы
   - **Цели**: измеримые цели
   - **Истории пользователя**: пользовательские сценарии
   - **Функциональные требования**: конкретные требования (пронумерованные)
   - **Вне области**: что не входит в задачу
   - **Оценка сложности**: простая/средняя/сложная/критичная (из Phase 0)
   - **Анализ кодовой базы**: найденные паттерны и компоненты (из Phase 2)
   - **Архитектурные решения**: выбранные подходы и обоснование
   - **План реализации**: этапы с критериями приемки
   - **Критерии приемки**: проверяемые критерии для каждого этапа
   - **Метрики успеха**: как измерять успех
   - **Открытые вопросы**: нерешенные вопросы
2. Сохрани PRD в файл `.tasks/task-[feature-name]/prd.md`
3. **Представь PRD пользователю для оценки**
4. **Жди явного подтверждения перед переходом к Phase 4**

**НЕ ПЕРЕХОДИ К PHASE 4 БЕЗ ЯВНОГО ОДОБРЕНИЯ ПОЛЬЗОВАТЕЛЯ**

## Phase 4: Проектирование архитектуры

**Цель**: спроектировать архитектуру реализации

**АДАПТИВНЫЙ ПОДХОД** - количество агентов зависит от сложности:

### Простая задача (1 агент):
- Запусти 1 агента `1c-architect`
- Создай один практичный план реализации

### Средняя задача (1 агент + опциональное мультисэмплинг):
- **По умолчанию**: 1 агент с практичным планом
- **Опционально**: если решение неочевидно, можешь запустить 2 агента с разными подходами и выбрать лучший

### Сложная/критичная задача (2-3 агента, мультисэмплинг):
- Запусти 2-3 агента `1c-architect` параллельно с разным фокусом:
  - Минимальные изменения (наименьшее изменение, максимальное переиспользование)
  - Чистая архитектура (поддерживаемость, элегантные абстракции)
  - Прагматичный баланс (скорость + качество)
- Сформируй мнение о лучшем подходе
- Представь пользователю все подходы с твоей рекомендацией
- **Спроси пользователя, какой подход он предпочитает**

**Действия**:
1. Запусти агентов `1c-architect` согласно сложности задачи:
   - Передай пути к артефактам: `.tasks/task-[feature-name]/phase1-requirements.md`, `.tasks/task-[feature-name]/phase2-exploration.md`, `.tasks/task-[feature-name]/phase3-clarifications.md`, `.tasks/task-[feature-name]/prd.md`
   - Для каждого агента укажи архитектурный подход (минимальные изменения / чистая архитектура / баланс)
2. После выбора подхода создай файл `.tasks/task-[feature-name]/phase4-architecture.md` с:
   - Постановкой задачи
   - Оценкой сложности из Phase 0
   - Выбранным архитектурным подходом с обоснованием
   - Найденными паттернами и соглашениями из Phase 2
   - Ответами на уточняющие вопросы из Phase 3
   - **ЭТАПАМИ РЕАЛИЗАЦИИ** - разбей реализацию на логические этапы (чеклист в markdown):
     - **Гранулярность**: простая задача = 1 этап, средняя = 2-4, сложная = 4-8, критичная = 5-10+ (отталкивайся от атомарности)
     - **Атомарность**: этап завершается за 1 сессию агента, имеет проверяемый результат, не блокирует сам себя
     - **Формат**: `- [ ] **Этап N**: описание + файлы + критерии приемки + зависимости (если есть)`
   - Mermaid-диаграммами (архитектура, потоки данных)
   - План должен быть самодостаточным для понимания без контекста беседы

## Phase 5: Ревью плана

**Цель**: валидировать план ДО реализации — ошибки в плане обходятся дорого, весь код придётся переписывать.

**Действия**:
1. Запусти агента `1c-arch-reviewer` для ревью плана:
   - Передай пути к артефактам: `.tasks/task-[feature-name]/phase1-requirements.md`, `.tasks/task-[feature-name]/phase2-exploration.md`, `.tasks/task-[feature-name]/phase3-clarifications.md`, `.tasks/task-[feature-name]/phase4-architecture.md`, `.tasks/task-[feature-name]/prd.md`
   - Фокус ревью:
     - **Полнота**: все ли аспекты задачи покрыты?
     - **Корректность**: соответствуют ли решения найденным паттернам из Phase 2?
     - **Реалистичность**: можно ли реализовать план как описано?
     - **Соответствие требованиям**: покрывает ли план все требования из Phase 1 и ответы из Phase 3?
     - **Критерии приемки этапов**: достаточно ли чёткие и проверяемые?
     - **Зависимости**: правильно ли выстроена последовательность этапов?
     - **Технический долг**: есть ли известные проблемы, которые план усугубит?
2. Если агент находит проблемы:
   - Обнови план в `.tasks/task-[feature-name]/phase4-architecture.md`
   - **Повтори Phase 5** (ревью обновлённого плана)
3. Когда план одобрен агентом, сохрани результаты ревью в `.tasks/task-[feature-name]/phase5-plan-review.md`:
   - Что было проверено (полнота, корректность, реалистичность и т.д.)
   - Какие проблемы были найдены и исправлены (если были)
   - Итоговое заключение: план готов к реализации

## Phase 5.5: Создание таск-листа

**Цель**: создать детальный таск-лист на основе архитектурного плана

**Действия**:
1. На основе архитектурного плана из Phase 4 создай таск-лист по следующей структуре:
   - **Relevant Files**: список файлов, которые будут созданы или изменены
   - **Tasks**: иерархический список задач с подзадачами

2. Структура таск-листа:
```markdown
## Relevant Files

- `path/to/file1.bsl` - Описание назначения файла
- `path/to/file.xml` - Описание формы объекта
- `path/to/another/file.bsl` - Описание вспомогательных функций

## Tasks

- [ ] 1.0 Родительская задача
  - [ ] 1.1 Подзадача 1.1
  - [ ] 1.2 Подзадача 1.2
- [ ] 2.0 Родительская задача
  - [ ] 2.1 Подзадача 2.1
```

3. **Правила создания таск-листа**:
   - Каждая родительская задача должна иметь чёткое описание
   - Подзадачи должны быть атомарными и проверяемыми
   - Подзадачи должны логически следовать из родительской задачи
   - Relevant Files должен содержать все файлы, которые будут затронуты
4. Сохрани таск-лист в файл `.tasks/task-[feature-name]/tasks.md`
5. **Представь таск-лист пользователю для оценки**
6. **Жди явного подтверждения перед переходом к Phase 6**

**НЕ ПЕРЕХОДИ К PHASE 6 БЕЗ ЯВНОГО ОДОБРЕНИЯ ПОЛЬЗОВАТЕЛЯ**

---

## Phase 6: Реализация по этапам

**Цель**: автоматическое выполнение задач из таск-листа

**НЕ НАЧИНАЙ БЕЗ ЯВНОГО ОДОБРЕНИЯ ПОЛЬЗОВАТЕЛЯ** (отдельное подтверждение получено в Phase 5.5)

### Алгоритм выполнения

1. Прочитай файл таск-листа `.tasks/task-[feature-name]/tasks.md`
2. **Для каждой незавершённой родительской задачи последовательно**:
   a. Найди все незавершённые подзадачи внутри неё
   b. **Для каждой подзадачи последовательно**:
      i. Отметь подзадачу в таск-листе как начатую (измени `- [ ]` на `- [O]`)
      ii. Запусти агента `1c-developer` для реализации подзадачи:
         - Передай путь к таск-листу: `.tasks/task-[feature-name]/tasks.md`
         - Передай путь к архитектурному плану: `.tasks/task-[feature-name]/phase4-architecture.md`
         - Передай путь к PRD: `.tasks/task-[feature-name]/prd.md`
         - Укажи конкретную подзадачу для реализации
         - Напомни про критерии приемки
      iii. Проверь критерии приемки подзадачи:
         - Все файлы созданы/изменены как описано?
         - Критерии приемки выполнены?
         - Код соответствует правилам из `project_rules.mdc`?
      iv. Если критерии не выполнены:
         - Запусти агента `1c-developer` для исправления:
           - Передай путь к таск-листу и архитектурному плану
           - Опиши найденные проблемы и что нужно исправить
         - Повтори проверку критериев
         - Повторяй цикл исправления, пока критерии приемки не будут выполнены
      v. Отметь подзадачу в таск-листе как завершённую (измени `- [O]` на `- [x]`)
      vi. Обнови файл таск-листа
   c. **После завершения ВСЕХ подзадач** внутри родительской задачи:
      - Отметь родительскую задачу как завершённую (измени `- [ ]` на `- [x]`)
      - Обнови файл таск-листа
   d. Если у родительской задачи нет подзадач — выполни её как самостоятельную единицу:
      - Отметь как начатую (`- [O]`)
      - Запусти `1c-developer`
      - Проверь критерии приемки
      - Отметь как завершённую (`- [x]`)

### Пример порядка выполнения

```markdown
## Tasks

- [ ] 1.0 Создать справочник Контрагенты
  - [ ] 1.1 Создать метаданные справочника        ← выполняется первой
  - [ ] 1.2 Добавить реквизиты                     ← выполняется второй
  - [ ] 1.3 Создать форму списка                   ← выполняется третьей
- [ ] 2.0 Настроить права доступа
  - [ ] 2.1 Создать роль                           ← выполняется четвёртой
```

**Важно**: Родительская задача 1.0 отмечается `[x]` только после того, как 1.1, 1.2 и 1.3 все отмечены `[x]`.

## Phase 7: Ревью кода

**Цель**: убедиться, что код корректен, элегантен и **соответствует PRD, таск-листу и архитектурному плану**

**Действия**:
1. Запусти ревьюеров согласно размеру задачи:
   - **Простая**: `1c-code-reviewer`
     - Передай путь к PRD: `.tasks/task-[feature-name]/prd.md`
     - Передай путь к таск-листу: `.tasks/task-[feature-name]/tasks.md`
     - Фокус: базовая корректность + проверка соответствия PRD и таск-листу
   - **Средняя/сложная**:
     - `1c-architect` для проверки соответствия плану:
       - Передай пути к артефактам: `.tasks/task-[feature-name]/prd.md`, `.tasks/task-[feature-name]/phase4-architecture.md`, `.tasks/task-[feature-name]/tasks.md`, `.tasks/task-[feature-name]/phase5-plan-review.md`
       - Фокус: соответствие плану, архитектурным решениям, критериям приемки этапов
     - `1c-code-reviewer` для проверки качества кода:
       - Укажи путь к правилам: `.cursor/rules/project_rules.mdc`
       - Передай путь к PRD: `.tasks/task-[feature-name]/prd.md`
       - Фокус: баги, читаемость, правила, DRY, элегантность, соответствие PRD
   - **Очень большая**: раздели код на логические модули/подсистемы, для каждого модуля запусти пару агентов (architect + reviewer) с теми же инструкциями
2. Консолидируй их ответы и выяви проблемы, требующие исправлений
3. Сохрани результаты ревью в `.tasks/task-[feature-name]/phase7-code-review.md`:
   - Что было проверено (соответствие PRD, таск-листу, плану, качество, баги и т.д.)
   - Найденные проблемы с severity и рекомендациями
   - Итоговая оценка готовности кода
4. **Представь выявленные проблемы пользователю и спроси, что он хочет сделать**:
   - Исправить сейчас
   - Исправить позже (добавить в TODO/технический долг)
   - Продолжить как есть
5. **Если "исправить сейчас"**:
   - Добавь проблемы в таск-лист в секцию "Исправления после ревью"
   - Запусти агента `1c-developer` для исправления:
     - Передай путь к таск-листу: `.tasks/task-[feature-name]/tasks.md`
     - Передай путь к PRD: `.tasks/task-[feature-name]/prd.md`
     - Опиши проблемы из ревью и что нужно исправить
   - **Вернись к началу Phase 7** (повторное ревью)
   - Повторяй итерации, пока ревьюеры не перестанут находить критические проблемы
6. **Если "исправить позже"**:
   - Добавь проблемы в таск-лист в секцию "Технический долг"
   - Отметь в таск-листе как отложенные

---

## Phase 8: Итоги

**Цель**: документировать выполненное

**Действия**:
1. Отметь все задачи как завершённые
2. Сохрани резюме в файл `.tasks/task-[feature-name]/phase8-summary.md`:
   - Что было построено (ссылки на этапы реализации)
   - Ключевые принятые решения (ссылка на план)
   - Изменённые файлы (git diff --stat)
   - Оставшийся технический долг (если есть)
   - Предлагаемые следующие шаги
3. **Представь резюме пользователю**:
   - Краткое резюме итога
   - Оставшийся технический долг (если есть)
   - Предлагаемые следующие шаги
4. **Спроси явно: "Реализация выполнена. Внести изменения в документацию?"**

**НЕ ПЕРЕХОДИ К PHASE 9 БЕЗ ЯВНОГО ОДОБРЕНИЯ ПОЛЬЗОВАТЕЛЯ**

---

## Phase 9: Изменение документации проекта

**Цель**: Обеспечить актуальность документации проекта

**Условие запуска**: Phase 9 выполняется **только** если:
1. Phase 8 уже завершена (есть `phase8-summary.md` и пользователю задан вопрос о документации), **и**
2. В **текущем ответе пользователя** есть явное подтверждение на обновление документации (например: «да», «внести изменения», «обновить документацию», «да, обновить»).

**Обязательная проверка перед Phase 9**:
- Если ты только что завершил Phase 8 в этом же ответе — **не переходи к Phase 9**. Остановись после вопроса «Внести изменения в документацию?» и дождись следующего сообщения пользователя.
- Если пользователь ответил на Phase 8, но без явного согласия на документацию — **не выполняй Phase 9**. Напомни итог Phase 8 и повтори вопрос: «Внести изменения в документацию? (напишите «да» или «обновить документацию» для перехода к Phase 9)».
- Выполняй Phase 9 **только** когда в последнем сообщении пользователя есть явное подтверждение.

**Действия** (только при выполненном условии запуска):
1. Запусти агента `1c-doc-writer`:
   - Передай путь к резюме: `.tasks/task-[feature-name]/phase8-summary.md`
   - Передай путь к PRD: `.tasks/task-[feature-name]/prd.md`
   - Опиши, что было сделано и попроси внести изменения в файлы документации
   - Дополни файл `.tasks/task-[feature-name]/phase8-summary.md` информацией об изменении документации
2. Представь резюме пользователю
3. Напомни: Phase 9 выполняется только после явного подтверждения пользователя; без подтверждения — не запускать

## Артефакты скилла

### Структура директории `.tasks/task-[feature-name]/`

```
.tasks/task-[feature-name]/
├── phase0-complexity.md          # Оценка сложности
├── phase1-requirements.md        # Требования и уточнения
├── phase2-exploration.md         # Исследование кодовой базы
├── phase3-clarifications.md      # Уточняющие вопросы и ответы
├── prd.md                        # Product Requirements Document
├── phase4-architecture.md        # Архитектурный план
├── phase5-plan-review.md         # Ревью плана
├── tasks.md                      # Таск-лист
├── phase7-code-review.md         # Ревью кода
└── phase8-summary.md             # Итоги
```

### Формат PRD (`prd.md`)

```markdown
# PRD: [Название задачи]

## Введение/Обзор
[Описание задачи и проблемы]

## Цели
- [Цель 1]
- [Цель 2]

## Истории пользователя
- Как [тип пользователя], я хочу [действие], чтобы [выгода]

## Функциональные требования
1. [Требование 1]
2. [Требование 2]

## Вне области
- [Что не входит]

## Оценка сложности
[Простая/Средняя/Сложная/Критичная]

## Анализ кодовой базы
[Найденные паттерны и компоненты]

## Архитектурные решения
[Выбранные подходы и обоснование]

## План реализации
- [ ] Этап 1: [Описание]
- [ ] Этап 2: [Описание]

## Критерии приемки
[Проверяемые критерии для каждого этапа]

## Метрики успеха
[Как измерять успех]

## Открытые вопросы
[Нерешенные вопросы]
```

```markdown
## Relevant Files

## Tasks

  - [ ] 1.2 Подзадача 1.2
- [ ] 2.0 Родительская задача
  - [ ] 2.1 Подзадача 2.1
```

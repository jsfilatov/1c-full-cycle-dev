## Инварианты

- **Полный цикл новой доработки**: фазы **0→12** последовательно, без пропуска **4–8** под видом «простая задача». Сложность влияет на глубину, число субагентов и **gate ADR после Phase 6**, не на набор фаз.
- **Исключение**: пользователь явно стартует по **файлу замечаний приёмки** и в `.tasks/task-[feature-name]/` уже есть артефакты исходной задачи — **не** повторять 0–8; выполнять раздел «Приёмочные исправления» (по сути **Phase 9 + 10** + обновление записей приёмки/итога).
- **Phase Gate**: переход только после явного ответа в **чате** на обязательную точку (не Plan mode Cursor, не «закрыто правкой markdown»). Не объединять несколько gate в один вопрос. При «делай как считаешь нужным» всё равно артефакт + формальное подтверждение там, где gate обязателен.
- **Заморозка кода**: при `режим_conveyor` = `strict`, пока не закрыт gate Phase 8 (или не `fast_track`, не режим приёмки) — оркестратор **не** меняет `src/cf/**`, `src/cfe/**` (только `.tasks/**` и чтение).
- **Субагенты (strict)**: при доступности не подменять полным самописом **`1c-analytic`** (PRD), **`1c-architect`** (план, ADR), **`1c-arch-reviewer`** (ревью плана); при недоступности — стоп, запись в `workflow-state.md`/чат, явное согласие на ручной артефакт.
- **Phase 9**: при доступности вызова каждая подзадача, меняющая `src/cf/**` или `src/cfe/**`, — **только** `1c-developer` или `1c-metadata-manager`, один вызов на подзадачу, в брифе **только** путь к `phase9-context-index.md`. Иначе — стоп и явное согласие на отступление. `fast_track` **не** отменяет это без записи о недоступности + согласия пользователя.
- **Артефакты `.tasks/`** — часть скилла (не общее «не трогай markdown»). **`acceptance-record.md`** создаётся в Phase 8, без отдельного gate; статусы прогона — **пользователь**.
- **Лимиты итераций**: цикл Phase 7 (п. 1–2) и цикл Phase 10 «исправить сейчас → ревью» — **≤ 10**; исчерпание → зафиксировать, эскалация пользователю.
- **Якорь фазы** в ответе пользователю: `**Фаза N/12:** … **Следующий шаг:** …` (в режиме приёмочных исправлений первая строка: `**Режим: приёмочные исправления.** **Следующий шаг:** …`).

## Приоритет инструкций и `fast_track`

- Фразы «реализуй всё», «не останавливайся», «complete todos» **не снимают** gate. **Стоп**, один выбор пользователю: **(A) strict** или **(B) fast_track**. Пока нет ответа или opt-in — **не** править `src/cf/**`, `src/cfe/**`.
- **`fast_track`** только после явной формулировки пользователя (смысл: включаю fast_track, gate Phase 1–8 пропускаю, ответственность на мне + дата/эквивалент). В `workflow-state.md`: `режим_conveyor` = `fast_track`, `fast_track_основание`, `fast_track_дата`, `ожидается_gate` = `—`, фаза — по указанию (часто 9). В `phase11-summary.md` указать аудитом, что конвейер был в `fast_track`.
- **Запрещено** закрывать gate правкой markdown без ответов в чате на **каждый** обязательный gate (в т.ч. не ставить `фаза_следующей_работы` ≥ 9 при strict до gate Phase 8 в чате).

| 7 | Результаты ревью; при изменении `adr.md` в цикле — см. Phase 7 п.3 | Gate ревью плана; при сработавшем условии ADR — **отдельное** сообщение, не смешивать с gate ревью |

| 11 | `phase11-summary.md` + вопрос о документации | Phase 12 только при **явном** согласии на документацию в **следующем** сообщении после вопроса |

### Дословные gate-вопросы

Задавать **один** вопрос за раз, **дословно** (допускается сопроводить кратким контекстом, но не менять смысл и не объединять gate).

**Phase 1** (отдельно от gate PRD): в исходном тексте скилла **не** была зафиксирована одна дословная фраза; требование — **отдельный** вопрос пользователю с резюме понимания и явным подтверждением перед Phase 2 (не смешивать с gate Phase 4).

**Phase 4 — утверждение PRD:**

> PRD готов. Утверждаете документ для перехода к проектированию архитектуры? (напишите "да" для подтверждения)

**Phase 5 — утверждение архитектурного плана:**

> Архитектурный план готов. Утверждаете план для фиксации решений в ADR? (напишите "да" для подтверждения)

**Phase 6 — утверждение ADR** (только если оценка **не** «Простая»; для «Простая» отдельного вопроса нет): представить `adr.md` и перечень минимальных правок `prd.md`, **остановиться** и дождаться **явного** утверждения ADR (например: «да», «утверждаю ADR») **до** Phase 7 — **без** единой фиксированной цитаты-шаблона в исходном скилле.

**Phase 7 — утверждение ADR после правок в цикле ревью** (только если `adr.md` менялся за цикл Phase 7; **отдельное** сообщение, не в одном с gate ревью плана):

> Утверждаете ADR после правок по ревью?

**Phase 7 — утверждение результатов ревью плана:**

> Ревью плана завершено. Утверждаете результаты для перехода к созданию таск-листа? (напишите "да" для подтверждения)

**Phase 8 — утверждение таск-листа:**

> Таск-лист готов. Утверждаете план реализации? После подтверждения начнётся автоматическое выполнение задач без дополнительных вопросов. (напишите "да" для подтверждения)

**Phase 11 — решение об обновлении документации:**

> Реализация выполнена. Внести изменения в документацию?

**Phase 12** — если пользователь ответил на Phase 11 без явного согласия на документацию, **повтор** приглашения к переходу в Phase 12:

> Внести изменения в документацию? (напишите «да» или «обновить документацию» для перехода к Phase 12)

**Правила gate (сжато):** один gate — один вопрос; не переходить без ответа; при возобновлении нового чата — опираться на `workflow-state.md` или **одним** вопросом уточнить открытый gate (нельзя выводить закрытие gate только по наличию файлов). Шаблон `workflow-state.md`: `references/workflow-state-template.md`. Идентификаторы `ожидается_gate`: `phase1_understanding`, `phase4_prd`, `phase5_architecture`, `phase6_adr`, `phase7_adr_after_review`, `phase7_plan_review`, `phase8_tasks`, `phase11_documentation`.

**Поддержка `workflow-state.md`:** создать в Phase 0; при старте — `strict` и пустые `fast_track_*`; после фаз без обязательного gate на выходе — обновить `фаза_следующей_работы`; **сразу после** сохранения артефакта, после которого в чате висит gate, — выставить `ожидается_gate`; **сразу после** явного ответа в чате — снять gate и выставить следующую фазу; отразить ветвления Phase 6–7 (в т.ч. доп. gate ADR после правок в Phase 7); при приёмочном режиме / `fast_track` — одной строкой в примечании и по правилам opt-in; при отсутствии файла при возобновлении — восстановить осторожно по артефактам, **одним** вопросом уточнить gate, затем создать/обновить файл.

## Phase 9 — Реализация

**Чеклист перед первым изменением прикладных исходников:** `workflow-state.md` — либо strict с закрытыми gate 1–8, либо `fast_track` с заполненными полями, либо приёмочные исправления; в strict — наличие PRD через `1c-analytic` при доступности агента или явное согласие на исключение.

**Исполнитель подзадачи:** строка **`Исполнитель:`** в индексе = ровно `1c-developer` | `1c-metadata-manager`, совпадает с `## Исполнители Phase 9` в `tasks.md`. **`1c-metadata-manager`** — мутации структуры метаданных XML (каталоги объектов, формы как Form.xml, СКД, MXL, роли, подсистемы, CFE-структура и т.д., см. `@.cursor/agents/metadata-manager.md`); читает скилл из ячейки индекса. **`1c-developer`** — остальное (BSL, логика при неизменной согласованной структуре). Метаданные+код — **последовательные** подзадачи, если иное не из ADR.

**Политика эскалации Phase 9:** субагенты **не** задают вопросы конечному пользователю; при блокере — стоп подзадачи, отчёт оркестратору; эскалация к пользователю — только оркестратор (противоречие tasks/adr/prd, нет объекта, права, продуктовый скоуп).

**Опционально `1c-error-fixer`:** только при исполнителе `1c-developer` и ошибках синтаксиса BSL; **≤ 3** вызова на подзадачу; не для `1c-metadata-manager`. Цикл исправления подзадачи: **≤ 10** попыток (включая error-fixer + основной исполнитель).

**Алгоритм:** перед каждым вызовом обновлять **только** `phase9-context-index.md` (текущая подзадача, выдержки, исправление); субагенту передать **только** путь к индексу. Порядок: секция **`## Tasks`**, затем **`## Приёмочные исправления`** если есть. Для каждой родительской задачи: подзадачи по порядку; в `tasks.md` маркеры `- [ ]` → `- [O]` → `- [x]`; после всех подзадач — родитель `- [x]`. Родитель без подзадач — как одна единица с тем же контрактом. Оркестратор **не** подменяет субагентов правками репозитория при доступности Task.

## Phase 10 — Ревью кода

Оркестратор собирает `phase10-code-review.md` из ответов субагентов.

**Ветвление:** «Простая» в phase0, но **> 7** путей в Relevant Files **или** пути из **≥ 2** типов корней метаданных (`Catalogs/`, `Documents/`, `InformationRegisters/`, `AccumulationRegisters/`, `AccountingRegisters/`, `DataProcessors/`, `Reports/`, `Enums/`, `Constants/`, `DefinedTypes/`, `CommonModules/`, `Roles/`, `ChartsOfAccounts/`, `ChartsOfCharacteristicTypes/`, `Subsystems/`) — ветка **средняя/сложная** (пара `1c-arch-reviewer` + `1c-code-reviewer`). Иначе «Простая» — только `1c-code-reviewer` с путями к `prd.md`, `tasks.md`, **`adr.md`**, **`phase5-architecture.md`**, `project_rules.mdc`. Средняя/сложная: архитектурный пост-аудит + код-ревью; разделение фокусов как в оригинале (архитектор — traceability и ADR; код-ревьюер — BSL, безопасность, стиль без полного дублирования traceability при уже вызванном арх-ревьюере). Сложная/критичная + большой объём — модульные пары ревьюеров по областям.

**Бриф `1c-arch-reviewer` в Phase 10:** в описании агента может быть «до реализации» — **здесь игнорировать**: пост-реализационный аудит соответствия утверждённым артефактам и diff/списку путей; не переоткрывать gate прошлых фаз; не предлагать новую архитектуру без явного противоречия коду и `adr.md`; обязательный выход — traceability tasks ↔ ADR ↔ изменения с severity.

Опционально: **`1c-performance-optimizer`** — один узкий вызов при NFR перформанса в prd/adr или HIGH+ от ревьюеров; **`1c-tester`** — по явному запросу или готовому deploy_and_test; **`1c-architect`** — узкая оценка при критичном расхождении с ADR (без переписывания кода в этом вызове).

Действия: запустить ревьюеров по ветке; консолидировать; спросить пользователя: исправить сейчас / позже / как есть. «Исправить сейчас» — секция в tasks или отдельные строки + тот же контракт Phase 9 через индекс; затем **с начала Phase 10**; **≤ 10** полных циклов.
